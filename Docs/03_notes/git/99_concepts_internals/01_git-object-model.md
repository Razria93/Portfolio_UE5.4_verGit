#### A1. Blob

##### 원본 키워드
- 파일 내용(바이트) 그대로 저장하는 객체
- 내용 기반 해시 (내용이 같으면 같은 해시)
- 트리/커밋은 전부 Blob을 조합해서 스냅샷 구성
##### 확장 설명
- Blob은 `.git/objects` 폴더에 저장되는 “파일 1개”에 해당하는 순수 데이터 덩어리임
- `git add` 시에 index의 hash와 현재 파일의 hash를 비교하고 hash값이 다를 경우 새로운 Blob을 만듦
- 해시는 `내용 → 해시 방식`이라, 내용을 한 글자라도 바꾸면 다른 해시가 됨
- 내용이 완전히 같으면 어느 커밋에서든 같은 blob을 재사용함

#### A2. Tree
##### 원본 키워드
- 디렉터리 1개에 대한 스냅샷
- “이름 → blob / tree 해시” 목록
- 하위 폴더는 하위 Tree로 연결

##### 확장 설명
- Tree는 디렉터리 구조를 표현하는 객체
- `git commit` 시에 파일들의 경로 및 폴더트리를 기반으로 파일과 폴더를 `그룹화`함
- 하위 `Level`부터 해당하는 `Blob`과 `하위 Folder`의 `hash`를 저장하고 해당 Folder의 hash를 만듦
- 이를 Root까지 재귀적으로 반복함
- 한 Tree 안에 :
	- 파일 이름 → Blob 해시
	- 하위 폴더 이름 → 하위 Tree 해시
- 최상위 디렉터리(/)에 대한 Tree 하나가 “그 커밋의 프로젝트 상태”를 대표하게 됨 (?)

---
#### A3. Commit

##### 원본 키워드
- 프로젝트 상태 스냅샷 + 이력 연결 정보
- 필수 필드 :
	- tree (루트 Tree 해시)
	- parent (부모 커밋 해시들)
	- author, committer, message

##### 확장 설명
- Commit은 “그 시점의 전체 파일 구조(Tree) + 과거 커밋과의 연결”을 담는 노드임
- Commit은 직접 Blob을 들고 있지 않고, Tree → Blob 구조를 통해 실제 파일 내용에 도달함

##### 연결 예
	Commit → Tree → Blob

---
#### A4. Ref (Branch / Tag)

##### 원본 키워드
- Ref는 `메모리주소에 대한 변수명`과 같이 `커밋 해시에 대한 또다른 이름`임
	- 예: `refs/heads/main`, `refs/tags/v1.0.0`

##### 확장 설명
- Ref는 커밋 해시를 기억하는 단순한 이름
- 필요에 따라 같은 `CommitHash`값이라도 다르게 부르기 위함임
- 브랜치: 개발 라인 (예: `main, dev, feature/login`)
- 태그: 특정 커밋에 단 이름 (`Release Version` 등)

실질적으로는 **“이 ref 파일에 어떤 커밋 해시가 적혀 있냐”**만 중요함

---

#### A5. HEAD
##### 원본 키워드
- “지금 내가 보고 있는 기준점”
- 보통: HEAD → 브랜치 ref → 커밋
- Detached HEAD: 직접 커밋 해시를 가리키는 경우

##### 확장 설명
- HEAD는 우리가 작업 중인 “현재 위치”를 나타냄
- 보통은 HEAD가 "`ref: refs/heads/main`" 같은 식으로 브랜치를 가리키고, 그 브랜치가 실제 커밋 해시를 가리키는 구조

##### 연결 예
 `HEAD → refs/heads/main → Commit C1234...`
